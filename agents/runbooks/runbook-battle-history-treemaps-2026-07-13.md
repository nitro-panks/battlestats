# Runbook: Battle-History Treemaps + Ship Population Avg-Damage Baseline

_Created: 2026-07-13_
_Context: Player-page Battle History gained three mini-treemaps above the sparkline (type / ships-by-damage / tier); the damage map's coloring required a new per-ship population baseline that must never be computed on the request thread._
_QA: 164/164 backend battle-history tests (sqlite), 4/4 new frontend component tests, live-verified locally on Flareside (warm path) and Domino_s_Pizza (cold path, previously a gunicorn worker-kill)._

## Purpose

Documents the 3.1.0 battle-history treemap feature: what the three maps encode, the new `ship_pop_avg_damage` payload field and `X-Ship-Pop-Pending` header, and the lazy-warm architecture that keeps the per-ship population aggregate off the request thread. Read this when touching `BattleHistoryTreemaps.tsx`, the battle-history payload shape, or the ship-population warm task.

## Decisions

- **Three maps, one area rule.** `BattleHistoryTreemaps.tsx` renders above the sparkline in `BattleHistoryCard`, fed the resolved `by_ship` rows (purely presentational — the card owns the fetch), so the maps always mirror the selected Window/Mode and the table below. AREA = battles everywhere (since 3.2.7). Layout (3.3.0): **two rows** — the ships map full-width on row 1 (it has the most tiles and earns the room), Type × WR + Tier × WR sharing row 2 at half width each AND half height (75px vs 150px, via a `MiniTreemap` `height` prop; mobile stacks all three). Titles follow a "size-dimension × color-signal" scheme.
  - **battles × dmg** (the ships map, row 1) — one tile per ship sized by **window battles** (3.2.7; originally total window damage, briefly avg damage — battles won as the size metric because it matches the other panels' volume semantics and the color already carries damage), colored on a **diverging scale** by player avg damage ÷ ship population average (red ≤0.6 · neutral gray 1.0 · green ≥1.5, Lab-interpolated, clamped). No baseline → neutral `#6f7683`. Clicking a ship tile toggles the same ShipStats panel as a table-row click. Scope (3.3.0): a **zoom slider** (1 → played-ship count; `.bh-scope-slider` slim track + 10px thumb in `globals.css`, bare-number `1 ↔ N` legend, right-aligned in the panel header) replaces the discrete Top-N | All toggle — the treemap re-lays-out live on every tick. Defaults to **min(25, roster)**; the chosen N persists per-browser in localStorage `bs-bh-ships-slider` (plain number, clamped to each player's roster on read; the pre-3.3.0 `bs-bh-ships-scope` key is orphaned). Since 3.2.3 the per-ship table's **Avg dmg column** reuses this exact scale (`damageRatioColor` is exported from `BattleHistoryTreemaps.tsx` and consumed by `AvgDamageCell` in `BattleHistoryCard.tsx`), so table text and treemap tiles always agree on above/below expectation; no-baseline rows fall back to plain strong text with an explanatory tooltip.
  - **Type × WR / Tier × WR** (row 2, half-width + half-height each) — DD/CA/BB/CV/SS (resp. tiers) sized by battles, colored by the group's aggregate WR (wins ÷ battles across its ships, `wrColor`). (A tier×type per-class clustered variant was tried on the 3.2.7 branch and reverted same-day.)
  - **Tile sub-line** — the value alone: avg dmg at 3 significant digits (`d3.format('.3~s')`) on the ships map, WR% on type/tier. Drawn only when it fits whole; the tooltip always carries the battle counts and full detail. (3.2.7 shipped a "career prior + window increment" battles prefix; 3.3.0 removed it as noise — a career-WR-delta third line was also tried and dropped pre-3.2.7.)
- **Baseline convention.** `ship_pop_avg_damage` is the realm-wide average damage over the trailing 30d of **random** battles (`SHIP_COMBAT_WINDOW_DAYS`, same population/window convention as the ShipStats panel), regardless of the view's mode — it is an expectation anchor, not a mode-scoped stat. Ships with under `SHIP_POP_AVG_MIN_BATTLES` (20) population battles carry `null` (a 0 sentinel in cache marks "computed, below floor" so it is never re-queued).
- **Never computed inline.** The first implementation ran one bulk `GROUP BY ship_id` population query on the request thread; a popular-T10 ship set (Domino_s_Pizza, 38 ships) took ~15s for 5 ships and **SIGKILLed the gunicorn worker** — the card errored and the page read as "no activity" while the (independently served) Vermont badge still showed. The shipped architecture mirrors the ship-list WR-percentile lazy warm:
  - `data.py`: `compute_ship_pop_avg_damage` (one small aggregate per ship, task-side only), `get_cached_ship_pop_avg_damage` (read-only bulk probe), cache key `ship_pop_avgdmg:v1:<realm>:<ship>:<day>` (26h TTL backstop on a day-scoped key).
  - `tasks.py`: `warm_ship_pop_avg_damage_task` (default lane, skip-if-cached) + `queue_ship_pop_avg_damage_warm` (per-ship `cache.add` dispatch dedup, 15 min).
  - `views.py`: `_attach_ship_pop_avg_damage` attaches from cache only, **per-request after the payload cache** (the payload cache never stores baselines, so warmed values appear without waiting out `BATTLE_HISTORY_CACHE_TTL`), queues the warm for misses, and returns pending → the response sets `X-Ship-Pop-Pending: true`.
  - `BattleHistoryCard.tsx`: on the pending header, re-polls up to 10× at 3s (`SHIP_POP_PENDING_*`, degradation-scaled, cacheBust rides the existing ranked-pending mechanism) — tiles colorize progressively as baselines land; stragglers stay neutral until a later visit.

## 2026-07-13 follow-up: nightly bulk pre-warm (all ships)

The day-scoped cache key meant every ship's baseline rotated cold at UTC midnight, so each day's **first** viewer of every ship ate the lazy warm (gray tiles → 3s poll → colorize). Now `snapshot_ship_top_players_task` (02:00+ UTC, per-realm striped — safely after the key rotation) chains `warm_all_ship_pop_avg_damage_task` (`background` queue, fail-open try/except around the enqueue), which calls `data.compute_all_ship_pop_avg_damage(realm)`: **one** grouped `GROUP BY ship_id` scan over the trailing 30d random PDSS computes + `set_many`s every ship's baseline, including the 0 below-floor sentinels (~34s/realm measured on prod data, ~900 NA ships — vs seconds *per ship* through the request-driven path). A "warm only T8-10" variant was considered and rejected: the grouped scan costs the same regardless of tier cut, T8-10 ships dominate the row count anyway, and viewers' cards contain ships of every tier. This is the same bulk query shape that SIGKILLed gunicorn **inline** (above) — the difference is it runs task-side on the background worker. The per-ship lazy warm + pending header + client poll all remain as the fallback for the midnight→snapshot gap and any stragglers.

## Follow-ups

- The 10×3s client poll covers only the first ~30s of a large cold set; a player opened once and never revisited keeps neutral tiles for the stragglers. Largely mooted by the nightly bulk pre-warm (above) — only viewers in the midnight→02:00+ UTC gap still exercise it.
- Prod first-day behavior: every realm starts cold, so expect a burst of warm tasks on the default lane after deploy; per-ship dedup caps the fan-out. If the default lane shows contention, move the task to the `background` queue.
